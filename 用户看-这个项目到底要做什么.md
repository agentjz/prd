# 用户看 - 这个项目到底要做什么

你现在要做的，不是开始写代码，也不是开始做技术方案。你现在要先把“这个项目到底要做什么”写清楚，而且这份文档是写给我看的，不是写给工程师看的。这份文档应该出现在“新开项目前准备 - 澄清需求”之后，也就是说，你已经通过提问，帮我把模糊需求逼清楚了一些；现在你要把这些已经澄清出来的东西，整理成一份我能直接看懂、直接判断方向对不对的项目说明。

1. 这是一份用户审阅版文档。我不看代码细节，也不靠技术术语理解项目。我主要看的是这个项目到底是什么，为什么要做，它真正解决什么问题，它是给谁用的，第一版最重要的结果长什么样，哪些事情这次不做，以及我怎么判断方向对不对。所以你写这份文档时，只用自然语言，尽量不用技术黑话，不要写成技术设计文档，不要上来讲数据库、框架、模块、接口，也不要把实现想法误写成需求本身。重点要写目标、用户、场景、结果和边界。

2. 这份文档至少要真正回答这些问题：这个项目到底是什么，它不是什么，它解决的核心问题是什么，这个问题对谁来说真的痛，为什么现在要做，如果第一版做出来最有价值的结果是什么，第一版最小成立状态是什么，哪些只是可选想法，哪些才是核心需求，市面上有没有可以直接参考、复用、改造的成熟东西，以及这次明确不做什么。你可以把这份文档理解成“立项前的项目定义文档”，但它是写给 Owner 看的，不是写给技术执行者看的。

3. 默认输出结构用 `1. 2. 3.` 的方式写成下面这些部分：1. 这个项目是什么；2. 为什么要做这个项目；3. 这个项目是给谁用的；4. 这个项目解决什么核心问题；5. 第一版最重要的结果长什么样；6. 第一版最小成立状态；7. 有哪些成熟参考可以借鉴或改造；8. 这次明确不做什么；9. 我作为 Owner 应该重点看什么。

4. 写作时要像在给不懂代码的人解释一个项目，不要空泛，不要口号化，每一段都要帮助我做判断。如果有地方还不确定，要明确写“待确认”，不要假装已经定了。这份文档的目标不是显得专业，而是让我真正看懂。如果你准备开始写，就基于我们前面澄清出来的需求，直接输出这份用户审阅版文档。
