# Agent使用 - 开始写SPEC文档

你现在不是来继续写一份中间层的 Agent review 文档的。我既然把这份提示词发给你，就表示前置的用户审阅版项目定义文档已经被我确认通过了。你现在不需要再反复确认这件事，你下一步就直接开始写正式的 SPEC 文档。

也就是说，你不要再额外写一篇“这个项目到底要怎么做”的过渡文章。你要直接把已经确认过的项目定义，落成真正能指导后续执行的 SPEC。正式的 spec 文档默认放在小写的 `spec/` 目录里，这里强调的是写文档规则，不是现在就要求你去搭复杂目录骨架。

前提是前置的用户审阅版项目定义文档已经被我认可，项目目标、用户、核心问题、第一版结果、范围外事项已经基本清楚，现在该进入正式写 SPEC 的阶段了。你必须始终记住这条主线：先把文档定死，再把测试定死，最后才写实现；你现在要写的 SPEC，不是空话文档，而是后续实现、测试、验证都会依赖的正式文档；SPEC 不能和已经确认过的用户审阅版项目定义冲突；SPEC 依然要分成两层，一层给我看，一层给你自己和其他 Agent 看。

但这里的“定死”不是一开始把所有细节写到极致完整。文档要轻量、清楚、够用，先定死目标、边界、主流程、验收和关键约束，细节可以随着开发推进继续补，但不能让根本目标和边界飘掉。你不要再额外输出第三种“过渡型 Agent review 文档”，你现在要做的，就是直接开始写 SPEC。

写 SPEC 时，你必须包含这些核心原则：文档先行，测试先行，实现后置，边界清晰，单一职责，小步推进，结果可验证，文档、测试、实现三者一致，文档保持轻量并且边开发边更新，测试优先抓主要矛盾和根本性漏洞。你还要主动考虑这个项目有没有成熟产品、成熟开源项目、成熟模板、成熟方案可以直接参考、复用或改造；如果有，就写进 SPEC 里，如果没有，再明确说明为什么要自己做。

默认输出结构如下。第一部分是用户审阅版 SPEC：1. 这个项目这一版到底要做什么；2. 这一版为什么值得做；3. 这一版是给谁用的；4. 核心场景和主流程；5. 验收标准；6. 范围外事项；7. 成熟参考与可复用对象；8. 待确认事项。第二部分是技术实现版 SPEC：1. 实现目标；2. 术语和边界；3. 实现前必须先定死的文档；4. 实现前必须先定死的测试与验收；5. 模块拆分；6. 每个模块的职责与非职责；7. 可复用、可改造、可参考的成熟方案；8. 推荐执行顺序；9. 验证与收口方式；10. 风险与待确认事项。

每一部分都要具体，不能空话，要能直接指导后续行动。用户审阅版不要被技术细节淹没，技术实现版要真正能指导后续实现；如果有不确定项，要明确标成“待确认”，不要硬编；如果有成熟轮子可以改，就优先写清楚怎么借鉴，不要默认从零造；如果某部分还不足以进入实现，要明确写出来，不要假装可以直接开工；文档不要过重，不要把还不需要的细枝末节提前塞满。

关于测试与验收，你必须特别遵守这些原则：测试要先抓核心，不要先抓细枝末节；优先定义会导致项目方向跑偏、结果失真、主流程失败、关键约束失守的失败测试；优先保护主路径、关键边界、关键状态、关键验收条件；不要一上来把测试写成过度脆弱的细节锁死；除非业务本身明确要求，否则不要把测试细化到一些无关主目标的名字、措辞、顺序、文案碎片这种级别；失败测试不是为了制造工作量，而是为了防止 AI 在根本问题上跑偏；要抓主要矛盾，抓核心思想，抓根本性漏洞，而不是被次要细节拖住；如果某个细节不影响项目核心目标、核心流程和核心验收，就不要过早把它写成刚性失败测试。

你在写 SPEC 时，要默认后续执行顺序是：1. 先完成用户审阅版 SPEC；2. 再完成技术实现版 SPEC；3. 再补齐失败测试和验收测试；4. 再开始最小实现；5. 再独立验证；6. 最后收口并同步文档。你的目标不是再解释一遍项目，而是直接把项目写进正式 SPEC，让后续 Agent 不用猜。如果你准备开始写，就基于已经确认过的用户审阅版项目定义，直接输出正式的 SPEC 文档，分成“用户审阅版 SPEC”和“技术实现版 SPEC”两部分。
