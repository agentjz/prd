# 一个文件只做一件事

这份文档只讲一件事：一个文件只做一件事，到底是什么意思。它不是形式主义，也不是为了好看而硬拆文件；它真正要保护的是低耦合、高清晰度、可维护性和后续长期演进的可靠性。简单不是保守，简单是可靠性和后续可维护性的前提；用 INTP 架构师的视角看，一个文件最重要的不是行数好不好看，而是它的职责边界是不是清楚，它为什么会变化，是不是只有一个主要原因。

1. 如果一个文件只承载一个清晰职责，而且读起来顺、改起来稳、边界也清楚，那就不要为了“拆得高级”而硬拆。单文件不等于坏，坏的是一个文件同时承担很多互相缠绕的职责。

2. 如果一个文件里同时混着多种不同性质的事情，比如状态管理、规则计算、数据读写、渲染展示、导入导出、外部接线、错误兼容、业务判断这些东西已经开始互相缠住，那就该拆。拆分不是为了文件更少行，而是为了让每一块东西都只围绕一个中心职责变化。

3. 不要为了拆而拆。不要把本来紧密相关、放在一起反而更清楚的东西，强行拆成一堆碎片文件；这样不会更优雅，只会让人到处跳来跳去，看起来低耦合，实际上更难理解。该拆的是耦合，不是字数。

4. 也不要为了图省事，把本来应该独立的逻辑全塞回一个大文件里。高耦合代码最典型的问题不是“文件大”，而是改一个地方要连带理解很多不相关的东西，一个小改动会把别的行为一起带崩，最后谁都不敢动。

5. 判断一个文件是不是该拆，可以看几个信号：它是不是有多个独立变化原因；是不是经常为了一个小改动就要读很多无关段落；是不是经常在同一个文件里同时改业务规则、状态流转和外部接线；是不是已经开始依赖大量注释来解释“这里为什么这么怪”；是不是任何一个函数改动都容易影响别的函数。如果这些信号越来越多，通常就不是继续忍的问题，而是结构已经该拆了。

6. 真正好的拆分结果，不是文件变多，而是边界更清楚。拆完以后，每个文件都应该能用一句人话说清“它负责什么，不负责什么”；模块之间尽量通过清晰数据和清晰接口连接，而不是互相偷拿状态、互相知道太多内部细节。

7. 最后记住一句话：不要强行拆，也不要硬塞回去；要做的是低耦合、高内聚、职责清楚、修改成本稳定的代码。如果一个文件只做一件事，就保留它；如果一个文件已经在做很多件事，就把它拆开。
