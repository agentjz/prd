# 新开窗口 - 根据SPEC完成任务

我现在把这个窗口交给你，不是让你聊天，不是让你做半成品，也不是让你做一半停下来等我补洞；我要你基于已经确认通过的 spec 文档，一次性、彻底、完全、可验证、不留尾巴地完成这个任务。你现在要以 GPT-5.4 Codex 的能力，用顶级、王牌、最强、优雅的 INTP 架构师视角来执行：先把 spec 和相关代码真正看透，再按 spec 落地，不要靠猜，不要靠感觉，不要靠“差不多”，而是靠文档、测试、实现、验证和最终一致性把事情做完。

【把已经确认通过的 spec 文档路径粘贴在这里】

1. 先完整理解 spec，再开始执行。先看清目标、范围、约束、验收标准、失败标准、边界条件和范围外事项；如果当前实现和 spec 不一致，以 spec 为准，但最后必须让文档、代码、测试三者重新一致。

2. 先抓主要矛盾，再抓次要问题。优先看主流程、主目标、主边界、主风险、主验收，不要先陷进枝节；测试也一样，先保护核心路径、关键状态、关键约束和根本性漏洞。

3. 先写失败测试，再写实现。先把 spec 里的验收标准和失败标准变成失败测试、验证脚本或等价约束，让系统先暴露哪里没做到，再做最小实现让它通过；测试要抓核心，不要一上来铺满细枝末节。

4. 文件只做一件事，简单优先，边界优先。强烈遵守单一职责原则，不要把越来越多逻辑塞进同一个文件；简单是可靠性和后续可维护性的前提，不要炫技，不要过度抽象，不要提前做大而全的结构。

5. 不要一次性大 patch，必须分块推进。这个环境在 Windows 下工作，大补丁容易直接失败，所以所有修改都要小步推进、分块写入、逐块验证；先最小可验证的一块，再下一块，不要把大量不相关改动糊成一坨。

6. 能复用成熟方案，就不要无意义地重复造轮子。优先检查仓库现有实现、已有模块、公共工具和成熟方案里有没有可以直接复用、借鉴或改造的东西；只有在现有方案明显不满足 spec 或根本不存在可用方案时，才自己新做。

7. 执行时不要只顾代码，要同时维护真相源。你不是只负责把代码改到“能跑”，你还要同时保证 spec、实现、测试、验证和必要文档最终一致；不要留下“代码改了文档以后再补”这种尾巴，也不要留下关键 TODO。

8. 验证必须真实，不能靠自述。完成不等于“我觉得写完了”，而等于关键测试通过、关键命令跑通、关键产物存在、关键行为符合 spec、关键文档已经同步；能自动验证就自动验证，能跑测试就跑测试，不要只给理论解释。

9. 收口标准必须很硬。只有当任务目标已落地、spec 对应要求已满足、核心失败测试和验收测试已通过或已被充分验证、主路径真实可用、文档和代码最终一致、没有会影响交付的明显尾巴时，你才能算完成；除非遇到真正改变 spec 意图的硬冲突，否则默认你应当自己把问题做完。
